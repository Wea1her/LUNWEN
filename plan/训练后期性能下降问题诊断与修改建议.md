# 训练早期收敛后性能下降问题的诊断与修改建议

## 问题描述

**症状**：训练很早就达到 best score，后面继续训练验证分数下降

**可能原因**：
1. 训练目标和验证目标不完全一致
2. 学习率略偏大，后期策略漂移

---

## 诊断分析

### 情况1：训练-验证目标不一致

**当前训练奖励结构**：
- 单步奖励：手续费 + 等待引导 + 风险惩罚
- 终止奖励：公平性 + starvation + 头尾风险 + Gas利用
- **主要驱动力**：手续费收益

**当前验证指标（two-stage）**：
- 第一阶段：可行性筛选（fairness ≥ 0.90, risk ≤ 0.30, edge10_risk ≤ 0.30）
- 第二阶段：可行域内按收益排序
- **主要驱动力**：可行率 → 可行域收益

**不一致点**：
- 训练时，模型学习"收益最大化"
- 验证时，模型评价"可行率优先 + 收益"
- 后期模型可能学会"高收益但边缘违约"的策略

### 情况2：学习率过大导致后期震荡

**当前学习率**：
```python
LR_ACTOR = 3e-4   # Actor 学习率
LR_CRITIC = 1e-3  # Critic 学习率
```

**问题**：
- 3000 episodes 训练，学习率恒定
- 前期需要较大学习率快速探索
- 后期需要较小学习率稳定收敛
- 恒定学习率导致后期"跳出"最优区域

---

## 解决方案

### 方案A：添加学习率衰减（首选，最简单）

#### 修改目标
在训练后期逐步降低学习率，使模型稳定在最优区域

#### 实现方式1：余弦退火（推荐）

**修改 `experiments/ppo.py`**：

```python
class PPOTrainer:
    def __init__(self, model: ActorCritic,
                 lr_actor=C.LR_ACTOR,
                 lr_critic=C.LR_CRITIC,
                 clip_eps=C.PPO_CLIP,
                 epochs=C.PPO_EPOCHS,
                 entropy_coef=C.ENTROPY_COEF,
                 device=torch.device("cpu"),
                 total_episodes=3000,           # 新增
                 lr_decay_enabled=True):        # 新增
        self.model = model
        self.clip_eps = clip_eps
        self.epochs = epochs
        self.entropy_coef = entropy_coef
        self.device = device
        self.lr_decay_enabled = lr_decay_enabled
        
        self.optimizer = torch.optim.Adam([
            {"params": model.encoder.parameters(), "lr": lr_actor},
            {"params": model.actor_fc.parameters(), "lr": lr_actor},
            {"params": model.actor_score.parameters(), "lr": lr_actor},
            {"params": [model.stop_embed], "lr": lr_actor},
            {"params": model.critic_fc.parameters(), "lr": lr_critic},
            {"params": model.critic_val.parameters(), "lr": lr_critic},
        ])
        
        # 新增：学习率调度器（余弦退火）
        if lr_decay_enabled:
            self.scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(
                self.optimizer,
                T_max=total_episodes,
                eta_min=lr_actor * 0.1  # 最小学习率为初始的 10%
            )
        else:
            self.scheduler = None
    
    def step_scheduler(self):
        """每个 episode 后调用，更新学习率"""
        if self.scheduler is not None:
            self.scheduler.step()
    
    def get_current_lr(self):
        """获取当前学习率（用于日志记录）"""
        return self.optimizer.param_groups[0]['lr']
```

**修改 `experiments/train.py`**：

在 PPOTrainer 初始化时添加参数：
```python
trainer = PPOTrainer(
    model, 
    device=device,
    total_episodes=args.episodes,
    lr_decay_enabled=True
)
```

在训练循环中，每个 episode 后调用：
```python
# PPO 更新
actor_loss, critic_loss, entropy = trainer.update(buffer)

# 新增：更新学习率
trainer.step_scheduler()
```

在验证日志中记录当前学习率：
```python
progress["current_lr"] = trainer.get_current_lr()
```

#### 实现方式2：分段衰减（更激进）

```python
def get_lr_multiplier(episode, total_episodes):
    """分段学习率衰减"""
    progress = episode / total_episodes
    if progress < 0.5:       # 前 50%：正常学习率
        return 1.0
    elif progress < 0.75:    # 50%-75%：降至 50%
        return 0.5
    else:                    # 75%-100%：降至 20%
        return 0.2

# 在 update() 中，每次更新前调整学习率
def update(self, buffer: RolloutBuffer, episode=0, total_episodes=3000):
    lr_mult = get_lr_multiplier(episode, total_episodes)
    for param_group in self.optimizer.param_groups:
        param_group['lr'] = param_group['lr'] * lr_mult
    # ... 原有更新逻辑
```

#### 实现方式3：线性衰减（最简单）

```python
class PPOTrainer:
    def __init__(self, ..., total_episodes=3000, min_lr_ratio=0.1):
        # ... 原有代码
        if lr_decay_enabled:
            self.scheduler = torch.optim.lr_scheduler.LinearLR(
                self.optimizer,
                start_factor=1.0,
                end_factor=min_lr_ratio,  # 最终降至初始的 10%
                total_iters=total_episodes
```

### 方案B：对齐训练和验证目标（更彻底，但复杂）

#### 修改目标
让训练奖励更加重视可行性，而不是纯收益

#### 方案B1：增加终止奖励中的可行性惩罚

**修改 `experiments/env.py`（或奖励计算部分）**：

```python
# 在终止奖励中加入可行性检查
def compute_terminal_reward(self, ...):
    # ... 原有计算
    
    # 新增：可行性惩罚
    fairness = self.compute_fairness()
    risk = self.compute_risk_exposure()
    edge10_risk = self.compute_edge10_risk()
    
    infeasible_penalty = 0.0
    if fairness < 0.90:
        infeasible_penalty += 1.0 * (0.90 - fairness)
    if risk > 0.30:
        infeasible_penalty += 1.0 * (risk - 0.30)
    if edge10_risk > 0.30:
        infeasible_penalty += 1.0 * (edge10_risk - 0.30)
    
    terminal_reward -= infeasible_penalty  # 减去可行性惩罚
```

#### 方案B2：改用 trade_score 作为验证指标

**修改训练命令**：
```bash
--val-metric trade_score  # 替代 two_stage
```

**优点**：
- 验证指标与论文评价指标一致
- trade_score 更综合，包含收益、公平、风险、效率

**缺点**：
- 可能导致模型忽视可行性约束
- 需要验证 trade_score 是否与训练奖励相关性更强

### 方案C：Early Stopping + Checkpoint 策略（最保守）

#### 修改目标
不修改训练过程，只是更智能地选择最优 checkpoint

#### 实现方式1：保存多个 checkpoint

**修改 `experiments/train.py`**：

```python
# 保存 top-K checkpoint
top_k_checkpoints = []  # [(score, episode, path), ...]
MAX_KEEP = 5

if current_score > best_score:
    # 保存新 checkpoint
    ckpt_path = f"top{len(top_k_checkpoints)+1}_ep{episode}.pt"
    save_checkpoint(ckpt_path)
    top_k_checkpoints.append((current_score, episode, ckpt_path))
    top_k_checkpoints = sorted(top_k_checkpoints, reverse=True)[:MAX_KEEP]
```

**最终评估时**：
- 从 top-5 checkpoints 中选择验证集上表现最好的
- 避免只用最后一个 checkpoint

#### 实现方式2：耐心策略（Patience）

```python
patience = 500  # 500 episodes 内没有提升就停止
patience_counter = 0

if current_score > best_score:
    best_score = current_score
    patience_counter = 0
else:
    patience_counter += 1
    
if patience_counter >= patience:
    print(f"Early stopping at episode {episode}")
    break
```


或者动态调整：

```python
def get_clip_eps(episode, total_episodes):
    """动态调整 PPO 裁剪范围"""
    progress = episode / total_episodes
    if progress < 0.5:
        return 0.2
    elif progress < 0.75:
        return 0.15
    else:
        return 0.1  # 后期更保守
```

## 推荐的组合方案

### 最小改动方案（推荐用于当前实验）

**组合**：方案A（余弦退火） + 方案C（top-K checkpoint）

**优点**：
- 代码改动最小
- 不影响训练奖励设计
- 快速验证是否有效

**实施步骤**：
1. 修改 `ppo.py`：添加 CosineAnnealingLR
2. 修改 `train.py`：每 episode 后调用 scheduler.step()
3. 修改 `train.py`：保存 top-5 checkpoint
4. 运行单 seed 实验验证

### 长期优化方案

**组合**：方案A + 方案B1（可行性惩罚） + 方案D（动态裁剪）

**优点**：
- 训练和验证目标更一致
- 后期更稳定
- 性能更优

**实施步骤**：
1. 先用方案A验证有效性
2. 如果仍有问题，再加入可行性惩罚
3. 微调 PPO 超参数
